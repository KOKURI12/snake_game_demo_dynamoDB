# Claude Code フロントエンド Token 節約ルール

Claude Code で HTML / CSS / JavaScript を扱う際、Claude が従うべき行動ルールをまとめる。
対象範囲を勝手に広げず、ユーザーが明示した範囲だけを調査・修正することで、token 消費と誤修正の両方を抑える。

---

## Claude が従う行動ルール

### 1. 対象範囲を勝手に広げない
- ユーザーが指定したファイル・関数・セレクタ・要素以外は読みに行かない
- 「ついでに周辺も確認」「関連ファイルも念のため」は **禁止**
- 指定が曖昧なときは推測せず、1行で「対象範囲を教えてください」と聞き返す

### 2. 原因調査と修正コード生成を混ぜない
ユーザーが「原因だけ」と言ったら、修正コードは出さない。
「修正コードだけ」と言ったら、長い解説は書かない。
明示されていなければ、まず **原因だけ** を先に返す。

### 3. 回答は短く、結論から
- デフォルトは **箇条書き3点以内**
- 結論 → 根拠 → 次のアクション の順
- 前置き・一般論・予防線(「状況によりますが」等)は書かない

### 4. 推測より「不足情報」
原因特定に情報が足りないときは、勝手に仮定を並べて長文化しない。
**不足している情報を箇条書きで返す** ことを優先する。

### 5. 提案を勝手に追加しない
- リファクタ提案
- 命名変更
- ベストプラクティス解説
- アクセシビリティ改善
- UI デザイン改善

これらは **明示的に求められたときだけ** 出す。聞かれていない提案は token の無駄。

### 6. 長文コード全文を要求しない
Claude 側から「該当ファイル全文を見せてください」と要求しない。
抜粋の範囲が不足しているなら、**何行目付近の何を追加で見たいか** をピンポイントで聞く。

---

## 調査の切り分け観点

不具合の種類で、見るべき箇所を絞る。

| 症状 | 最初に見る範囲 |
|---|---|
| レイアウト崩れ | 対象セレクタ / 親要素 / position・display・flex・grid・overflow / media query |
| レスポンシブ崩れ | media query / 該当幅での対象要素のみ |
| JS エラー | エラー行 ± 数行 / 発火イベントのハンドラ / DOM 取得部分 |
| クリック効かない | event listener / セレクタマッチング / preventDefault / z-index |
| 非同期不具合 | fetch / async-await / state 更新タイミング |

主因が HTML / CSS / JS のどこかを、回答の冒頭で明示する。

---

## 標準の応答フォーマット

原因特定を求められたとき:

```
主因: [HTML / CSS / JS のいずれか]
原因: [1〜2行]
根拠: [1〜2行]
次アクション: [最小修正の方向性を1行]
```

修正コードを求められたとき:
- 差分または修正後の該当箇所のみ
- 変更していない部分は再掲しない
- 解説は1〜2行まで

---

## ユーザー向けコピペプロンプト(厳選)

### A. 原因特定のみ依頼
```
原因特定だけしてください。修正コードは不要です。
見る範囲は以下に貼るコードのみ。回答は箇条書き3点以内。
```

### B. 最小修正コードのみ依頼
```
この関数/セレクタだけ、最小修正コードを返してください。
解説とリファクタ提案は不要です。
```

### C. 差分レビュー
```
この差分だけレビューしてください。
バグ可能性・崩れリスク・不要変更の3観点で、最大3点まで。
```

### D. セッション要約(新規セッションへの引き継ぎ用)
```
ここまでの対応を200文字以内で要約してください。
対象ファイル名・原因候補・未解決点を含めてください。
```

---

## 禁止事項

- 対象範囲外のファイルを勝手に読む
- 聞かれていない改善提案を追加する
- minify 済みコード・lock ファイル・build 出力を全文読み込む
- 1ターンで「調査 + 修正 + リファクタ + 解説」を全部やる
- 「より詳しく説明すると…」で長文化する

---

## 最重要原則

**対象範囲限定・原因先行・修正は後で。**

この3点を守れば token 消費と誤修正が大きく減る。
